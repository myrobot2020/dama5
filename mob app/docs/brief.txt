# DAMA App — Project Brief

## Project Name

**DAMA**
**Expanded form:** *Dhamma Alignment & Verification Architecture*

## One-line Summary

DAMA is an evidence-grounded AI system for Buddhist learning and guidance that answers only from verified canonical sources and trusted commentary, with full traceability, chain visibility, and minimal distortion.

## Core Idea

Most AI spiritual assistants are fluent but unreliable. They paraphrase too freely, mix traditions, flatten nuance, and often generate “wisdom-like” text that sounds right without being textually grounded.

DAMA is designed to solve that.

Instead of treating Dhamma as just content inside a chatbot, DAMA treats Dhamma as the **governing architecture** of the system itself. The canon is not decoration, not extra context, and not optional reference material. It is the fixed source of truth. The system retrieves, verifies, and responds from that source first, while commentary is used only in a controlled and auditable way.

---

# 1. Problem Statement

## Current problem in AI spiritual tools

Existing AI tools in this space usually have these weaknesses:

* They hallucinate or over-generalize.
* They merge canonical teaching with modern self-help language.
* They give answers without showing where the answer came from.
* They cannot distinguish between scripture, commentary, and generated explanation.
* They are optimized for smoothness, not fidelity.
* They do not teach progressively or structurally.

For a domain like Dhamma, this is a serious issue. Distortion is not a minor bug. It breaks the point of the teaching.

## Why this matters

Dhamma is a preservation-sensitive domain. A system that speaks about it casually can mislead users while sounding confident. That creates spiritual, educational, and ethical risk.

DAMA exists to build an AI system where:

* truth is grounded in source,
* answers are inspectable,
* commentary is clearly separated,
* and preservation is not sacrificed for convenience.

# 3. Product Goal

DAMA should help a user ask a question like:

* “What does the canon say about sloth and torpor?”
* “What are the seven powers?”
* “Show me related chains across the Anguttara Nikaya.”
* “What is the difference between the sutta and the teacher’s explanation?”

And instead of generating a loose answer, the system should return:

1. the relevant canonical passage,
2. the matched transcript segment,
3. the teacher commentary,
4. the chain or doctrinal structure involved,
5. and a final grounded answer assembled from those sources.

---

# 4. Core Product Principles

## 1. Canon is fixed

The canon is non-negotiable. It is the highest reference layer.

## 2. Evidence before generation

Every answer should be supported by source text, not model confidence.

## 3. Separation of layers

The system must clearly distinguish:

* canonical text,
* commentary,
* metadata,
* and generated explanation.

## 4. Traceability

Every answer should be auditable. The user should be able to inspect how the answer was formed.

## 5. Minimal distortion

Modern explanation is allowed only if it converges back to the source and does not alter meaning.

## 6. Structured teaching

The system should not only answer questions but also help users move through Dhamma in a gradual and structured way.

# 6. Current Corpus Focus

The current core focus is the **Anguttara Nikaya**, especially transcript-based teaching material associated with Bhante Hye Dhammavuddho Mahathera.

This includes:

* canonical references like **AN x.y**
* teacher transcript material
* commentary aligned to suttas
* cross-links to canonical sources such as SuttaCentral
* chain extraction from enumerated teachings

A major emphasis is on preserving mapping between:

* transcript numbering,
* teacher-specific splits,
* and canonical numbering systems.

This is important because the transcript structure does not always map cleanly onto standard canonical IDs.

# 7. What Makes DAMA Different

## Not just a chatbot

DAMA is not a normal RAG bot with religious documents stuffed into a vector DB.

## It is a verification architecture

Its core innovation is architectural:

* source-aware ingestion,
* sutta/commentary separation,
* canonical ID mapping,
* chain extraction,
* evidence-first retrieval,
* traceable response building.

## It answers with structure

DAMA is designed to understand doctrinal lists, causal chains, grouped teachings, and numbered structures rather than relying only on keyword similarity.

Example:
A normal system retrieves by phrase match.
DAMA should retrieve by **doctrinal structure** too.

# 8. Core Features

## A. Verified Q&A

User asks a question and receives:

* relevant sutta evidence,
* commentary evidence,
* canonical links,
* grounded final answer.

## B. Multi-pane evidence view

Proposed interface has 3 evidence panes plus final response:

1. **Transcript pane**
2. **Commentary pane**
3. **Canonical pane**
4. **Answer pane**

This lets the user see source layers separately.

## C. Chain extraction

DAMA identifies enumerated teachings such as:

* sets of 2,
* 5,
* 7,
* 11,
  and represents them explicitly.

This is especially useful in the Anguttara Nikaya, where numbered structures are fundamental.

## D. Chain graph / doctrinal graph

The system should show how items relate, including:

* chain length,
* ordering,
* doctrinal grouping,
* upstream/downstream connections.

## E. Canonical mapping

Transcript material is matched to canonical sutta identifiers and external canonical repositories.

## F. Commentary containment

Commentary can enrich the answer, but only when linked to a specific canonical basis.

## G. Transparent provenance

The user can inspect:

* which chunks were retrieved,
* why they were selected,
* which layer they came from,
* and how the final answer was assembled.

# 9. Technical Architecture

## High-level architecture

DAMA consists of several layers:

### 1. Ingestion Layer

Processes source material such as:

* transcripts,
* canonical text,
* commentary,
* metadata.

### 2. Structure Extraction Layer

Detects:

* sutta boundaries,
* commentary boundaries,
* numbered teachings,
* chain types,
* reference IDs.

### 3. Mapping Layer

Maps transcript segments to:

* canonical IDs,
* SuttaCentral IDs,
* cross-reference structures.

### 4. Retrieval Layer

Searches by:

* semantic similarity,
* doctrinal structure,
* canonical references,
* chain membership,
* metadata filters.

### 5. Verification Layer

Checks whether the answer is sufficiently grounded and whether commentary is properly aligned.

### 6. Response Assembly Layer

Builds a final answer using:

* canonical quotes,
* commentary support,
* traceable reasoning structure,
* explicit source references.

### 7. UI Layer

Displays:

* evidence panes,
* chain views,
* retrieval graph,
* final answer,
* system status.

This lets the system retrieve not just text, but structured doctrinal units.

# 11. Retrieval Philosophy

## Normal RAG problem

Normal RAG often chunks arbitrarily, mixes commentary with source, and retrieves based on shallow similarity.

## DAMA retrieval design

DAMA retrieval should be:

* page-aware,
* sutta-aware,
* commentary-aware,
* chain-aware,
* canon-aware.

It should avoid mixed chunks where sutta and commentary are blended.
Instead, it should split by structure first and then embed intelligently.

That is a major product advantage.

# 12. AI/Agent Design

Handles live user queries:

* retrieve,
